Howto

Intro

The idea is : the properties of a json data model shall have a direct conterpart into the html template, and vice versa. 

    //The applicative data model
    var shop = {
      ...
      books : [
        ...
        {
          ...
          buyers : [
          ...
          {
            name : "Albert"
          },
          ]
        }
        
      ]
    }
    //Instanciate the $scp.$dm datamodel object
    $scp.$Damo(shop,function() { ... });
    ...
    
    //$scp.$dm.books[4].buyer[6].name being available, the
    //template can use :
    <tagName id="books[4].buyer[6].name">...</tagName>

Get started

It is very simple to initialize Damo. You just have to call $scp.$Damo(dataModel[,prog]), where dataModel is either an url, either a JSON objet. In case of an url, the server that will return the data must send JSON data.
The optional prog is a function that will be executed after the first loading of the app. Typically this is where you manipulate the variables and functions needed to make your application alive, as doing somthing on a click for instance. 

<script>
  //Get a data model, either by calling an url :
  myDataModel = 'http://myhost.com/server/getData';
  //or by building it inside the script :
  myDataModel = { ... };

  //Then start Damo	 
  $scp.$Damo(myDataModel,function() {
    //here comes your code managing the app after loading it
    ...
  });			
</script>

Once loaded, the data model is available in the Damo's variable $scp.$dm for instance :

//data model
shop = {
  ...
  books : [
	...
	{
	  price : 123.4
	  ...
	}
  ]
}

//Damo's start
$scp.$Damo(shop,function() { ... });

//Damo's data model
$scp.$dm.books[3].price

Of course the script must have an html counterpart to display the properties of the data model. The two-way data binding is based on the ids of the html tags, that must correspond to the data model, for example :

//Damo's data model
$scp.$dm.books[3].price 
		
//html counterpart
<div id="books[3].price"></div>
Two-way data binding

Each time a property of the datamodel is changed in the template, with an input, a select or a textarea, the whole applicative data model $scp.$dm is automatically updated and the template view is updated to reflect the modification.
However when a property of the data model is changed in the script, a $scp.$updateView() must be launched for the template to reflect the change. This avoids a dirty checking to monopolize system resources and to cause asynchronous misbehaviors. Therefore the script -> template update has to be programmatically written as so : 

$scp.$updateView();

If the view update is necessary before inserting some code, the usage of $.when(...).done(function() {...}) is also necessary, to be sure to apply the code on the fresh view. As an example :

  $scp.promotion = setPromo($scp.$dm.books);
  $scp.$updateView().done(function() {

    $('input[id$="price"]').change(function() {
      console.log('-->change promotion price');
      $scp.$updateView();
    });
    $('select[id$="promotion"]').change(function() {
      console.log('-->change promotion status');
      $scp.promotion = setPromo($scp.$dm.books);
      $scp.$updateView();
    });
  });
Display the data model

The tags of the html template are connected to the applicative data model, and the variables of the scope, by their ids :

<tagName id="dataModelPropertyName">

In these expressions dataModelPropertyName is of type : books[4].buyer[6].name and is the counterpart of the property $scp.$dm.books[4].buyer[6].name of the datamodel.

There is however an exception for the option tag. Indeed This tag can not include any html tag, but just text. So we can not use something like this :

//bad way of settig up an option
<option value="books[4].buyer[6].name"><span id="books[4].buyer[6].name"></span></option>

In this case we use the {{...}} pattern like this :

//bad way of settig up an option
<option value="books[4].buyer[6].name">{{books[4].buyer[6].name}}</option>
Conditions

The display of an HTML element might be conditional, depending on a scope variable, function or a data model variable. The syntax to use is the following :

<tagName damo-if="condition">...</tagName>

The evaluation is made on the truth of the condition. This last may be the result of a function, an expression using ! == != < <= > >= ==, or the simple presence of a scope variable $scp.myVar. Some examples :

//corresponding to what the function $scp.myFunction() returns
<tagName damo-if="myFunction()">...</tagName>
//corresponding to $scp.myVar
<tagName damo-if="myVar">...</tagName>
<tagName damo-if="myVar >= 26">...</tagName>
<tagName damo-if="myVar != myOtherVar">...</tagName>
//corresponding to $scp.$dm.books[4].buyer[6].name
<tagName damo-if="books[4].buyer[6].name.length>20">...</tagName>
Loop

You might have lists to display, and furthermore lists with a changing length along the application, and that could be fastidious to code. Damo provides a tag attribute that will handle this kind of problem. Here is an example :

<ul>
  <li damo-loop="book in books" id="book.name"></li>
</ul>

In this example the li tag will be fed with $scp.$dm.books[i].name. Any sort of html tag can be repeated this way.

Trigger

This is an equivalent of the click() function of Jquery, except that only the name of the function is needed (no parenthesis). Here is an example :

<div damo-trigger="myTriggerFunction"></div>

This syntax will trigger the function when the html tag is clicked.

Filters

The values of the data model, and the variables of the scope, can be first filtered before being displayed. This is made by two ways :

  • by their ids :
    <tagName id="dataModelPropertyName" damo-filter="filterName">
  • by the connector expression :
    {{dataModelPropertyName | filterName}}

In these expressions filterName is the name of the function that will be used to filter the value, i.e. $scp.$filter.filterName().
The filters can only be a method of the object $scp.$filter, it can be coded inside the main script or included in a javascript file in the header of the index.html. Example :

$scp.$filter.numFilter = function(num) {
  num = parseFloat(num);
  return num.toFixed(2);
};
Routing

Damo is designed for single page applications, but as you know, this kind of applications must exhibit many pages. So Damo is equiped of a routing array that describe, for each page, the url to display in the browser"s address bar, and the url of the template. The routing array is a member of the scope $scp. It can be declared inside the main script or included from one (or many) javascript file in the header of index.html. It must respect the following format : 

$scp.$routing = [
  {
    url : '/myPage',
    template : 'templates/mypage.html'
  },
  ...
};

If this routing array is not empty, it is possible to change the page by calling : 

$scp.$changePage('/myPage');

Note that the back and forward buttons of the browser are then operationals, but a reload leads back to the home page.

Forms

A form must be enclosed into a specific tag, and have a trigger to get the key/value data. It must respect the following syntax :

//form tag
<div damo-form="myTriggerFunction">
...
</div>
//trigger of the form, could be inside the form tag
<button damo-formsubmit="myTriggerFunction">Go Submit</button>

The script counterpart must be a function $scp.myTriggerFuntion() in which the reserved array $scp.$submitData,instanciated with the key/value data of the form, is available as soon as the trigger has been fired. An example could be :

$scp.myTriggerFunction = function() {
    console.log($scp.$submitData);
};

Note that the click() trigger of the damo-formsubmit tag can not be modified, all the needed actions must be included to the damo-submitform function.

Directives

A directive is a non standard html tag that stands for an entire template. The directives might be included in the main index.html, with one or many files, respecting the $scp.$directive object format. This template shall have its own data model, i.e. all the ids of the elements must refer to a root object, called seed. This seed enables to connect with the main datamodel $scp.$dm, by yhe mean of the current template. For instance :

<div damo-loop="item in books">
  <myShop damo-seed="item"></shop>
</div>

To make a directive available, it shall be a member of the $scp.$directive object with the following format :

$scp.$directive = {
  directiveName : {
    seed : "directiveSeed",
    template : "pathToTemplate"
  },
  ...
}

Syntax

HTML template

Word Pattern Usage 
<tagName damo-key="value">
...
</tagName>
or for option tags : 
<option value="dataModelId">
  {{dataModelVariable}}
</option>
 This is the reference format for all the Damo's Html tags. The words in italic have to be replaced by the significant words.
damo 
<div damo-damo="url">
...
</div>
 This tag is made to enclose a page of the application. The url is described in the routing.js file as $scp.routing[i].url
filter 
<tagName id="varId" damo-filter="filterName">
...
</tagName>
or for option tags : 
<option value="varId">
  {{varId | filterName}}
</option>
 This pattern applies a filter to the concerned variable having the data model id varId, e.g. shop.books[2].promo.price.
form 
<tagName damo-form="functionName">
...
</tagName>
 This patterns enclose a form. Each element inside this tag will be considered as being a modifiable data that will have to be returned to the server, or whatever process. The function functionName will be triggered when the element <tagName damo-submitform="functionName" ...> is clicked, but before this the variable $scp.$submitData will be fed with the key/value data of the form.
loop 
<tagName damo-loop="arrayName">
...
</tagName>
 This pattern calls a loop base of the content of the array which name is arrayName. The tagName can be of varous type : div, td, tr, option, li, or anything else.
seed 
<directiveName damo-seed="seedName">
...
</directiveName>
 This pattern indicates which word will be considered as the root data of the directive (see Directives).
submitform 
<tagName damo-submitform="functionName">
...
</tagName>
 This pattern calls the function which name is functionName, when the concerned tagName is clicked. After the click the system first feeds the variable $scp.$submitData which contains all the key/value data contained in the form.
trigger 
<tagName damo-trigger="functionName">
...
</tagName>
 This pattern calls the function which name is functionName, when the concerned tagName is clicked.

Script

Pattern Usage 
$scp
 This is the main scope of Damo. As a good practice you can add any member or method to it, but try to keep the $scp.$xxxx pattern reserved for Damo. Prefer to use $scp.xxxx. All the variables and methods that you will need for your application shall be a member or method of $scp, in order to allow the two-way data binding.
Note that any method added to $scp becomes a service, at an angular point of view, but it is immediatly available to the whole scope, for all pages. 
$scp.$dm
 This is the reserved object that will contain the entire json data model of the application. 
$scp.$submiData
 This is the reserved array where is stored the key/value data of a form, as soon as the trigger function of the form have been fired. 
$scp.$Damo(json | url [,prog])
 This is the first expression to insert to your application script. It will feed the datamodel $scp.$dm with the provided json, either directly, or asking it to an url. The optional prog is a function that contains all the application code and that will be applied after the build of the first view. 
$scp.$updateView()
 The two-way data binding of Damo is not automatic, this avoid a dirty checking to monopolize system resources and to cause asynchronous misbehaviors. The view will be updated only if you ask for it. 
$scp.$changePage(pageUrl)
 This expression will change the current page for the required page. It might be called by clicking any tag, or inside a link with the following format : 
<a href="javascript:{$scp.$changePage('pageUrl')}">...</a>
 
$scp.$routing = [
  {
    url : "pageUrl",
    template : "pathToTemplate"
  },
  ...
]
 [optional Array] If you whish to setup a single page application, with many pages, this is how you shall provide the routing of the different pages. This might be done by including one (or many) javascript file in the index.html, containing the $scp.$routing array. The url is what will appear in the browser's address bar, the default being "/" if no routing have been given. The template is a string giving the path to the concerned template, e.g. myFiles/templates/myTemplate.html. The default template is "index.html" that must be present in case no routing is specified. 
$scp.$filter = {
  filterName : function(data) {
    //your code here ...
    return filteredData;
  },
  ...
}
 [optional Object] You might need filters to display your data. For instance you could whish to display an icone in case of a particular value. All filters are methods of the object $scp.$filter.You might provide the filters by including one (or many) javascript file in the index.html, containing the $scp.$filter object. There are two ways to use filters : 
<tagName id="varId" damo-filter="filterName">...</tagName>
or 
{{varId | filterName}}
 
$scp.$directive = {
  directiveName : {
    seed : "directiveSeed",
    template : "pathToTemplate"
  },
  ...
}
 [optional Object] A directive is a non standard html tag that stands for an entire template. The directives might be included in the main index.html, with one or many files, respecting the $scp.$directive object format. This template shall have its own data model, i.e. all the ids of the elements must refer to a root object, called seed. This seed enables to connect with the main datamodel $scp.$dm, by yhe mean of the current template. For instance : 
<div damo-loop="item in books">
  <myShop damo-seed="item"></shop>
</div>
